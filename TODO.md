# 交互节律 TODO

这个文件记录低频迭代路线。交互节律不是高速增长型产品，优先维护可信、稳定、可长期运行的小工具形态。

## 维护原则

- 更新频率保持低：优先按季度或重大 bug 触发发布，不追求周更。
- 每次发布只解决一个主主题，避免把稳定工具改成试验田。
- 数据安全优先于 UI 优化，UI 优化优先于新增解释能力，新增解释能力优先于提醒和自动化。
- 所有更新必须能从旧版本发现新版，并完成应用内更新。
- 不引入账号、云同步、数据回流，除非明确形成新的隐私设计和发布边界。

## 版本节奏

- 补丁版，例如 `1.7.2`：只修 bug、数据保护、更新链路、启动和退出稳定性。
- 小版本，例如 `1.8`：允许加入一个清晰的新能力，同时保留原有主页面结构。
- 大版本，例如 `2.0`：只有当数据模型、定位或跨端能力发生明显变化时再考虑。

建议节奏：

- 日常不主动发版。
- 每 1-2 个月整理一次问题池。
- 每 3-4 个月合并一次维护版。
- 发现更新链路、数据丢失、无法启动、无法退出这类问题时立即发补丁版。

## 发布检查表

每次发布前必须完成：

- [ ] Python 语法检查：`python -m py_compile config.py db.py main.py monitor.py settings.py stats.py tray.py update_manager.py window.py`
- [ ] 前端脚本语法检查：从 `static/index.html` 抽取 `<script>` 后用 Node 执行 `new Function(...)`。
- [ ] 本机 current 启动成功，`/api/today` 返回非错误结果。
- [ ] `/api/settings` 响应正常，设置页不阻塞。
- [ ] `/api/update/status` 快速返回，远端刷新不阻塞 UI。
- [ ] `/api/insights?days=7` 返回三条以内建议。
- [ ] `build.ps1` 成功生成 current、`交互节律.zip`、`interaction-rhythm.zip` 和 `update.json`。
- [ ] `build-installer.ps1` 成功生成安装器。
- [ ] 至少一个旧版本测试副本能发现新版。
- [ ] 至少一个旧版本测试副本能完成应用内更新。
- [ ] 带真实数据的测试副本更新后，`tracker.db` 关键计数不减少。
- [ ] GitHub Release 资产包含安装器、便携包和 `update.json`。
- [ ] 远端 `latest/download/update.json` 返回新版本。
- [ ] `docs/releases/v版本.md` 写入更新内容和验证结果。
