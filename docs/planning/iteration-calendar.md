# 交互节律迭代日历

交互节律采用低频维护节奏：稳定版本不追高频功能，优先处理真实使用里暴露的启动、更新、数据保留、后台行为和解释质量问题。

## 节奏

- 维护版：半月或一月检查一次，只处理阻塞体验和数据风险。
- 功能版：一个月左右推进一次，必须有明确用户价值。
- 方向评审：连续两到三版后做一次，判断是否进入更大的结构变化。
- 不为了排期强行发版；没有真实问题时记录 no-op。

## Super Productivity 同步

Super Productivity 目前通过任务 `dueDay` 进入日历。当前本地 REST API 能创建和更新任务，但没有暴露项目创建接口，所以采用以下规则：

- 任务标题统一使用 `[交互节律] 版本号 主题`。
- 任务备注第一行写 `项目：交互节律`。
- 到期日使用上表日期。
- 项目文件夹和长期上下文保存在 Poiesis 与本仓库，不依赖任务工具。

## 发布条件

- `README.md`、`CHANGELOG.md` 和对应 release note 已更新。
- `dist/releases/交互节律.zip` 不含数据库、日志、截图、缓存。
- GitHub Release 包含 `interaction-rhythm.zip` 和 `update.json`。
- 旧版本应用内检查更新能发现新版。
- 打包版启动、托盘退出、热力图、导出、节律助手和检查更新均通过冒烟测试。
